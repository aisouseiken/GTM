# 05. データモデル設計

DB: Supabase (PostgreSQL)。全テーブルに `workspace_id` を持たせ RLS でテナント分離。

---

## 1. ER図（概念）

```
users ──< workspace_members >── workspaces ──< api_keys
                                    │
                                    ├──< chat_sessions ──< chat_messages
                                    │
                                    ├──< search_plans ──< jobs ──< job_events
                                    │                        │
                                    │                        └──< leads ──< lead_verifications
                                    │                                  └──< lead_sources
                                    ├──< lead_lists >──< lead_list_items >── leads
                                    │
                                    └── credit_wallets ──< credit_transactions
subscriptions ── workspaces
```

## 3. 主要な設計判断

- **`search_plans` を独立保存**することで「同じ条件の再実行＝再現性」を実現（origamiの弱点対策）。
- **`lead_sources` / `lead_verifications` を分離**し、出典と検証根拠を透明化（信頼獲得）。
- **`cost_internal` をjobに記録**し、販売クレジットと原価の差＝粗利を常時監視。
- `signals` / `enrichment` は可変項目のため **jsonb** で柔軟に。検索は必要に応じてGINインデックス。
- 名寄せは `domain` / `phone` / 正規化住所をキーに。曖昧一致は pgvector で補助可能。

## 4. インデックス方針（抜粋）
- `leads(workspace_id, job_id)`, `leads(domain)`, `leads(workspace_id, status)`

- `credit_transactions(workspace_id, created_at)`
- `signals`/`enrichment` に GIN（jsonb検索する場合）
