# 04. システム設計書

## 1. アーキテクチャ全体像

```
┌─────────────────────────────────────────────────────────────┐
│                        クライアント                            │
│   Next.js App Router (React) + Tailwind + shadcn/ui           │
│   ・チャットUI（ストリーミング）・結果テーブル・課金画面           │
└───────────────┬─────────────────────────────────────────────┘
                │ Server Actions / Route Handlers (fetch, SSE)
┌───────────────▼─────────────────────────────────────────────┐
│                    アプリケーション層 (Next.js)                 │
│  ・認証(Supabase Auth)  ・APIルート(/api, /v1 公開API)          │
│  ・エージェント・オーケストレーター (Vercel AI SDK)              │
│  ・クレジット計測  ・ジョブ登録                                  │
└───────┬───────────────────────────┬─────────────────────────┘
        │ enqueue                    │ read/write (RLS)
┌───────▼──────────┐        ┌────────▼──────────────────────────┐
│  ジョブワーカー     │        │        Supabase (PostgreSQL)       │
│  (非同期・並列)     │◀──────▶│  users, workspaces, jobs, leads,   │
│  ・検索実行         │        │  search_plans, credits, api_keys … │
│  ・検証/エンリッチ  │        │  + pgvector（意味検索/名寄せ補助）   │
└───────┬──────────┘        └───────────────────────────────────┘
        │ adapter interface（抽象化・MOCK対応）
┌───────▼───────────────────────────────────────────────────────┐
│                    外部サービス・アダプタ層                       │
│  LLM(Claude) │ 検索/データソース │ メール/電話検証 │ Stripe │ CRM  │
│         ↑ すべて MOCK_MODE で差し替え可能（最後に実キー投入）        │
└───────────────────────────────────────────────────────────────┘
```

## 3. エージェント & コネクタ設計（コア）

### 3.1 オーケストレーターの流れ
```
ユーザー発話
  │
  ▼
[1] 意図解釈: LLM が自然言語 → 構造化ICP(JSON) に変換（Zodスキーマで検証）
  │      { industry, location, signals[], size, market, ... }
  ▼
[2] プランニング: どのコネクタを・どの順で・どのパラメータで叩くか決定
  │      → search_plan として保存（再現性の要）
  ▼
[3] （ユーザー承認）
  ▼
[4] 実行(ジョブ): コネクタを並列実行 → LeadCandidate[] を収集
  ▼
[5] 名寄せ・重複排除・正規化 → Lead[]
  ▼
[6] 検証・エンリッチ（ウォーターフォール）＋ 信頼度スコア付与
  ▼
[7] クレジット消費確定（成功分のみ）→ 結果保存 → ストリーム返却
```

### 3.2 コネクタ・インターフェース（抽象化）
```ts
// すべてのデータソースはこの契約を実装する
interface DataSourceConnector {
  id: string;                 // "google_maps" | "job_boards" | ...
  markets: Market[];          // ["JP","GLOBAL"]
  search(input: SearchInput): Promise<LeadCandidate[]>;
  costPerCall: CostHint;      // 原価見積り（クレジット計算に使用）
}

// 検証・エンリッチも同様に抽象化
interface Verifier {
  verifyEmail(email: string): Promise<VerificationResult>; // score付き
  verifyPhone(phone: string): Promise<VerificationResult>;
}
```
- **実装は `RealConnector` と `MockConnector` の2系統。** `MOCK_MODE=true` で後者を注入（DIコンテナ or ファクトリ）。
- 市場（JP/GLOBAL）でコネクタ集合を切替。

### 3.3 コネクタ例（市場別）
| 種別 | GLOBAL | JP |
|------|--------|-----|
| ローカル企業 | Google Maps系 | Google Maps系 + 地図/タウン系 |
| 求人シグナル | 求人検索API | Indeed系・国内求人 |
| 企業属性 | LinkedIn系企業データ | 法人番号API(gBizINFO等) |
| EC検出 | Shopify/Woo検出 | 国内カート検出 |
| SNS/広告 | X, 広告透明性 | X, 各広告透明性 |

> 実サービス名・キーは [06](./06_外部API連携設計.md) にプレースホルダで管理。**この段階では契約（interface）だけ確定させ、中身は最後に差す。**

## 4. 非同期ジョブ設計

- 検索は重い＆外部API律速のため**必ず非同期ジョブ化**。
- ジョブは冪等・再試行可能。外部API失敗時は部分結果を保存し `partial` で返す。
- プラン別の同時実行数（concurrency）をキューで制御（Starter=10, Pro=20, Scale=50 …）。
- 進捗はDBの `job_events` に追記 → クライアントへSSEで配信。

---

## 5. クレジット計測

- 各コネクタ/検証の `costPerCall` を合算し**事前見積り**を算出。
- 実行時は「成功したデータ取得・検証」に対してのみ消費を確定（トランザクションで残高更新）。
- 原価（外部API従量＋LLMトークン）をジョブ単位で記録 → 粗利モニタリング（NFR-7）。

---

## 6. セキュリティ / プライバシー

- Supabase RLS で `workspace_id` によるテナント分離を全テーブルに適用。
- 公開APIキーはハッシュ保存（表示は発行時のみ）。
- B2B連絡先に限定、オプトアウト・削除要求フローを用意（GDPR/個人情報保護法）。
- 監査ログ（誰がいつ何を取得/エクスポートしたか）。

---

## 7. 可観測性

- 構造化ログ（ジョブID・ワークスペースID・コネクタ・原価）。
- ジョブトレース（各ステップの所要時間・件数）。
- コストダッシュボード（外部API消費・LLMトークン・粗利アラート）。

## 8. 環境構成

| 環境 | 用途 | 外部API |
|------|------|---------|
| local | 開発 | **MOCK_MODE=true**（キー不要で全機能動作） |
| staging | 検証 | 一部実API + サンドボックス決済 |
| production | 本番 | 実API + 本番決済（最終フェーズ） |

主要な環境変数（`.env.example` に定義予定）:
```
MOCK_MODE=true
MARKET_DEFAULT=JP
SUPABASE_URL= / SUPABASE_ANON_KEY= / SUPABASE_SERVICE_ROLE_KEY=
LLM_API_KEY=            # 最後に設定
DATASOURCE_*_API_KEY=   # 最後に設定
VERIFY_*_API_KEY=       # 最後に設定
STRIPE_SECRET_KEY=      # 最後に設定
```
