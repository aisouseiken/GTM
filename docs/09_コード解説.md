# 09. コード解説（このコードはこういう意味です）

> 目的：エンジニアでなくても「どのファイルが何をしているか」が分かるように、
> 実装したコードを平易な日本語で説明します。専門用語には都度かんたんな注釈をつけます。

---

## 0. まず全体像（3行で）

1. ユーザーがチャットに「こんな顧客を探して」と**日本語で入力**する。
2. **AI役のプログラム**がその文章を読み解き「検索プラン」を作り、実行すると**モックの会社リスト**を生成する。
3. 生成した会社は**連絡先を検証**してスコアを付け、**表**に並べて表示・保存・CSV出力できる。

※ 今は外部サービス（本物のAI・データ会社・決済）に**まだ繋いでいない**ので、
その部分は「モック（＝本物そっくりの偽物）」で動かしています。最後に本物へ差し替えます。

---

## 1. フォルダの役割

| フォルダ | 何が入っているか | ひとことで言うと |
|----------|------------------|------------------|
| `app/` | 画面（ページ）と裏方の窓口（API） | 「見た目」と「入口」 |
| `app/api/` `app/v1/` | サーバー側の処理窓口 | 「裏方の受付」 |
| `components/` | 画面の部品（ボタン・表・チャットなど） | 「レゴのブロック」 |
| `lib/` | ロジック本体（AI解釈・検証・データ保存など） | 「頭脳と倉庫」 |
| `docs/` | 設計書・仕様書・この解説 | 「説明書」 |

---

## 2. 「頭脳と倉庫」= lib/ の中身

### `lib/domain/types.ts` — 用語の定義書
このコードは、アプリで扱う**データの形（項目）を決めている**だけのファイルです。
たとえば「リード（見込み客）には、会社名・ドメイン・メール・電話・信頼度…という項目があります」
という“決めごと”を書いています。**プログラム全体の共通言語**になります。

### `lib/data/store.ts` — データの倉庫
このコードは、**データを一時的に覚えておく倉庫**です。
ユーザー・ワークスペース・リード・クレジット残高などを、パソコンのメモリ上に保管します。

- 「今はメモリ（＝一時記憶）なので、サーバーを再起動すると消えます」。
- 設計書どおり、**あとで本物のデータベース（Supabase）に差し替えられる**形にしてあります。
- `spendCredits()` は「残高が足りればクレジットを引く」という**お財布の出し入れ**の関数です。

### `lib/agent/planner.ts` — AIの“読み解き役”
このコードは、**日本語の文章から「検索の条件」を取り出す**部分です。
例：「東京の歯科医院で採用中」→ 業種＝歯科、地域＝東京、シグナル＝採用中、と分解します。

- `INDUSTRIES` は「この単語が入っていたらこの業種」という**辞書**です。
- 今は辞書ベースの簡易AIですが、`planWithLLM()` という**本物のAIに差し替える差込口**を用意済み。

### `lib/mock/leadgen.ts` — 会社リストの“生成役”
このコードは、**それらしい会社のリストを作り出す**部分です（本物のデータ会社に繋ぐ前の代役）。

- 日本向けなら「株式会社◯◯歯科」、海外向けなら「Cedar Dental」のような名前を作ります。
- 重要ポイント：**同じ検索なら毎回同じ結果**が出るように作っています（＝再現性。設計書の差別化ポイント）。
  「サイコロを振る」のではなく「決まった計算式」で作るのでブレません。

### `lib/agent/verify.ts` — 連絡先の“検証役”
このコードは、**メールや電話が本物っぽいかを何段階もチェックし、0〜100点の信頼度を付ける**部分です。

- メールなら「形式 → ドメイン存在 → 業者A → 業者B」と**滝のように順番に確認**（＝ウォーターフォール）。
  ある段階で十分な確度が出たら、そこで打ち切ってムダを省きます。
- **検証に成功したぶんだけ課金**する、という設計もここに入っています。

### `lib/agent/runner.ts` — 実行の“司令塔”
このコードは、検索プランを受け取って**一連の作業を順番に進める司令塔**です。

1. コネクタ（データ源）を順に検索 → 2. 重複をまとめる（名寄せ） → 3. 一件ずつ検証 → 4. 保存。
- 各ステップで「今なにをしているか」の**進捗メッセージを発信**します（画面左に出るあれ）。
- クレジットが尽きたら**途中まで（部分完了）**で止める、という気配りも入っています。

### `lib/auth/session.ts` — ログインの“受付”
このコードは、**簡易ログイン**の仕組みです。メールを入れるとブラウザに“合言葉（cookie）”を持たせ、
次からその人だと分かるようにします。今はパスワード不要のデモ用。最後に本物の認証へ差し替えます。

### `lib/auth/apikey.ts` — 外部連携用の“合鍵”発行
このコードは、**プログラムから使うためのAPIキー（合鍵）**を作る部分です。
発行時だけ実物を見せ、保管は**暗号化した痕跡（ハッシュ）だけ**にして安全にしています。

### `lib/config.ts` — 設定と便利道具
「モックで動くか」「日本/海外どちら既定か」などの**設定**と、金額・数値の**表示整形**の道具箱です。

## 3. 「入口」= app/api/ と app/v1/ の窓口

これらは画面ではなく、**画面の裏でデータをやり取りする受付**です（URLでアクセスされる）。

| ファイル | 役割（このコードはこういう意味です） |
|----------|--------------------------------------|
| `app/api/plan/route.ts` | 「この文章で検索プランを作って」の受付 |
| `app/api/jobs/route.ts` | 「そのプランを実行するジョブを用意して」の受付 |
| `app/api/jobs/[id]/stream/route.ts` | 実行しながら**進捗を少しずつ画面へ流す**窓口（＝SSE。実況中継のようなもの） |
| `app/api/leads/route.ts` | リード一覧の取得／お気に入り・除外の切替 |
| `app/api/lists/route.ts` | 検索結果を「リスト」として保存 |
| `app/api/export/route.ts` | リードを**CSV（Excelで開ける表）**で書き出し |
| `app/api/apikeys/route.ts` | APIキーの発行・一覧 |
| `app/v1/search/route.ts` | **外部プログラム向け**の検索窓口（合鍵が必要）※差別化機能 |
| `app/v1/jobs/[id]/route.ts` | 外部プログラム向けの結果取得窓口 |

## 4. 「見た目」= app/（ページ）と components/（部品）

### ページ（app/）
- `app/page.tsx` … **トップページ**（宣伝ページ）。大きな見出し・デモ表・料金など。
- `app/login`, `app/signup` … ログイン／新規登録の画面。
- `app/app/layout.tsx` … アプリ内の**共通の枠**。「未ログインなら入れない」門番の役割もここ。
- `app/app/w/[id]/page.tsx` … **メイン画面**（チャット＋結果表）。`[id]` はワークスペースごとに変わる番号。
- `app/app/w/[id]/billing` … 料金・クレジット残高・履歴。
- `app/app/w/[id]/api-keys` … APIキー管理と使い方。
- `app/app/w/[id]/leads` … 保存したリストの一覧。

### 部品（components/）
| ファイル | 役割 |
|----------|------|
| `Logo.tsx` | GTM のロゴ（折り紙の鶴をイメージしたピンクのマーク） |
| `MarketingNav.tsx` | トップページ上部のメニュー |
| `LandingDemo.tsx` | トップに置いた**アプリの見本**（チャット＋表） |
| `FitBar.tsx` | 適合度を示す**緑のバー**と、会社の頭文字アイコン |
| `AppShell.tsx` | アプリの**外枠**（左のアイコン列・上のバー・クレジット表示） |
| `Workspace.tsx` | **中心の画面**。入力→プラン→実行→表への反映を全部つかさどる（一番大事な部品） |
| `LeadDrawer.tsx` | 会社をクリックすると右から出る**詳細パネル**（検証の根拠や出典を見せる） |
| `ApiKeysPanel.tsx` | APIキーの発行ボタンと一覧 |

### とくに `components/Workspace.tsx` の流れ（このコードの心臓部）
1. ユーザーが文章を送る → `submitPrompt()` が裏方（`/api/plan`）に渡して**検索プラン**を受け取る。
2. 「この内容で実行する」を押す → `runPlan()` がジョブを作り、**実況中継（SSE）を購読**。
3. 実況で「1件見つかった」と来るたびに**表を更新**。「完了」で残高やステータスを整える。
4. 会社をクリックすると詳細パネル、星でお気に入り、保存やCSV出力もここから。

## 5. いま「モック（代役）」で、最後に「本物」に差し替えるところ

| 代役でやっていること | 最後に差し替える本物 |
|----------------------|----------------------|
| 会社リストを自作（`leadgen.ts`） | 本物のデータ会社API（地図・求人・企業DB等） |
| 文章解釈を辞書で（`planner.ts`） | 本物のAI（Claude等） |
| 連絡先チェックを計算で（`verify.ts`） | 本物のメール・電話検証API |
| メモリに保存（`store.ts`） | 本物のデータベース（Supabase） |
| 残高は数字だけ（`billing`） | 本物の決済（Stripe） |

**差し替えても画面や流れは変わりません**。各所を「差込口（インターフェース）」にしてあるので、
中身だけ本物に入れ替えれば動く——という作りにしています（設計書04の方針どおり）。

## 6. 動かし方（開発者向けメモ）

```bash
npm install     # 初回だけ：必要な部品をダウンロード
npm run dev     # 開発サーバー起動（http://localhost:3000 前後）
```
- 追加設定なしで動きます（`.env.example` を参照。今は `MOCK_MODE=true`）。
- 型チェック：`npx tsc --noEmit`（エラーが無いことを確認できます）。
