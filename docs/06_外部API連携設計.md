# 06. 外部API連携設計（★キー投入・決済は最終フェーズ）

> ご指示どおり、**外部APIの実キー設定と決済は最後**に行う。
> それまでは全て **アダプタ層 ＋ MOCK_MODE** で動作させ、実装・デモ・レビューを止めない。

---

## 1. 依存する外部APIの分類

| # | 分類 | 役割 | 実装優先 | キー投入 |
|---|------|------|---------|---------|
| A | LLM API | 意図解釈・プランニング・エージェント推論 | コア（早期にモック→比較的早く実接続でも可） | 最終前 |
| B | データソース/検索API | リード発見（地図/求人/企業/EC/SNS/広告） | コア | **最終** |

## 2. アダプタ層の設計（差し替え可能に）

すべての外部依存は interface 経由。実体は `Real*` と `Mock*` を用意し、環境変数で注入。

```ts
// 例: プロバイダ・レジストリ
function getConnectors(market: Market): DataSourceConnector[] {
  if (process.env.MOCK_MODE === "true") return mockConnectors(market);
  return realConnectors(market); // 実キー使用（最終フェーズで有効化）
}
```

### MOCK_MODE の要件（NFR-9）
- キー無しで**全画面・全フロー**が動く。
- モックは**決定的**（同入力→同出力）。再現性機能のテストにも使える。
- 現実的なダミー（日本語企業名・国内電話形式・海外企業名の両方）。

## 3. 候補プロバイダ（プレースホルダ）

> 実選定は最終フェーズ。ここでは「どの枠にどの候補を差すか」の設計のみ。
> 選定基準: カバレッジ / 単価（粗利）/ 規約・法令適合 / レート制限 / 日本対応。

### A. LLM
- 候補: Claude（Anthropic）を第一候補（推論品質・ツール利用）。
- 用途: 構造化ICP変換、プランニング、要約、名寄せ補助。
- コスト管理: トークン量をjobの `cost_internal` に加算。

### B. データソース / 検索（GLOBAL / JP）
| 枠 | GLOBAL候補 | JP候補 |
|----|-----------|--------|
| Web検索 | 汎用検索API | 同左 |
| ローカル企業 | 地図系API | 地図系API + 国内タウン系 |
| 求人シグナル | 求人検索API | Indeed系・国内求人 |
| 企業属性 | LinkedIn系データプロバイダ | gBizINFO / 法人番号API（公式・無料枠あり） |
| EC検出 | Shopify/Woo検出系 | 国内カート検出 |
| 広告出稿 | 各広告透明性ライブラリ | 同左 |

> ⚠️ スクレイピング系は**規約・法令順守が前提**。公式API・許諾のあるデータプロバイダを優先。gBizINFO等の公的オープンデータは日本市場の強力な武器。

### C. 検証（ウォーターフォール）
- メール: syntax→MX→複数プロバイダ（到達性/リスク判定）。
- 電話: フォーマット正規化→キャリア/有効性判定プロバイダ。
- 詳細アルゴリズムは [08](./08_精度向上戦略.md)。

### D. エンリッチ
- 技術スタック検出、トラフィック推定、企業規模推定 等（プロバイダ差し替え可）。

### E. 決済（Stripe）
- サブスク（プラン）＋ 従量（追加クレジット購入）。
- Webhookで支払い状態を `subscriptions` に同期。
- **最終フェーズで実装**。それまではUIの骨組みとダミー残高で代替。

### F. CRM
- HubSpot / Salesforce へ OAuth 連携し、選択リードを書き出し。
- 差別化ポイント（origamiが弱い領域）。

## 4. 公開REST API（自社が提供する側）

差別化として最初から設計に含める（origamiには公開APIが無い）。

```
POST /v1/search
  body: { prompt: string, market?: "JP"|"GLOBAL", max_results?: number }
  → { job_id }

GET  /v1/jobs/{job_id}
  → { status, result_count, credits_spent, leads: [...] }

POST /v1/webhooks         # ジョブ完了通知の登録
```
- 認証: `Authorization: Bearer <api_key>`（`api_keys` テーブルでハッシュ照合）。
- レート制限・クレジット消費は内部フローと共通。

---

## 5. コスト & レート制御

- 各プロバイダの `costPerCall` と rate limit をレジストリに定義。
- 重複呼び出しを避けるためキャッシュ（同一ドメインの検証結果など）。
- ジョブ単位で原価を集計し、**販売クレジット単価 > 原価** を監視（アラート）。
